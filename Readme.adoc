== System Architecture & Design

The service decouples the critical path of link redirection from telemetry logging. This ensures that the user experience remains lightning-fast, bound only by database index lookup speeds ($O(1)$ amortized complexity), completely isolated from analytical ingestion writes.

=== Architectural Workflow

[mermaid]
----
sequenceDiagram
    autonumber
    actor User as Client Browser
    participant API as Spring Boot Controller
    participant DB as PostgreSQL Database

    activate API
    API->>DB: Query original URL (Indexed short_code)

    DB-->>API: Return Original URL Mapping
    deactivate DB

    Note over API,Async: Dispatch telemetry logging asynchronously
    API-->>Async: recordClickAsync(Mapping, IP, User-Agent, Referer)

    API-->>User: HTTP 302 Found (Location: target_url)
    deactivate API

    activate Async
    Async->>DB: INSERT INTO click_analytics
    deactivate Async
----

=== Key Engineering Decisions

* **Sequence-Based Base62 Generation:** Instead of using random hashes (which can cause hash collisions and require database-heavy retries), this system relies on a PostgreSQL sequence. The database sequence generates a unique `BIGINT` which is mathematically encoded to a base-62 string, guaranteeing $O(1)$ conflict-free code generation.
* **Non-Blocking Telemetry:** Analytical data capture (logging IP address, User-Agent, Referer, and Timestamp) is offloaded to a Spring `@Async` managed thread pool. This prevents slow database inserts from blocking the HTTP redirection thread, keeping response times under 15ms.
* **Regex-Constrained Routing:** To prevent the greedy catch-all endpoint (`/{shortCode}`) from hijacking static assets (like `/index.html` or `/favicon.ico`), the controller uses regex validation (`{shortCode:[a-zA-Z0-9\-_]+}`) to distinguish valid Base62 hashes from static file paths.

== Database Schema Design

The database contains two core tables managed via **Flyway Schema Migrations**:

=== Table: `url_mappings`
Stores the mapping between shortcodes and original destination URLs.

[cols="1,2,3", options="header"]
|===
| Column | Type | Constraints
| `id` | `BIGSERIAL` | `PRIMARY KEY`
| `original_url` | `TEXT` | `NOT NULL`
| `short_code` | `VARCHAR(50)` | `UNIQUE`, `NOT NULL`, `INDEXED`
| `is_custom` | `BOOLEAN` | `DEFAULT FALSE`, `NOT NULL`
| `created_at` | `TIMESTAMP WITH TIME ZONE` | `DEFAULT CURRENT_TIMESTAMP`, `NOT NULL`
|===

=== Table: `click_analytics`
Captures visit telemetry asynchronously.

[cols="1,2,3", options="header"]
|===
| Column | Type | Constraints
| `id` | `BIGSERIAL` | `PRIMARY KEY`
| `url_mapping_id` | `BIGINT` | `FOREIGN KEY` references `url_mappings(id)`
| `clicked_at` | `TIMESTAMP WITH TIME ZONE` | `DEFAULT CURRENT_TIMESTAMP`, `NOT NULL`
| `user_agent` | `TEXT` |
| `referer` | `TEXT` |
| `ip_address` | `VARCHAR(45)` | Supports IPv4/IPv6
|===

== API Contracts

=== 1. Shorten URL
* **URL:** `/api/v1/shorten`
* **Method:** `POST`
* **Request Payload:**
+
[source,json]
----
{
  "url": "https://www.google.com",
  "customAlias": "custom-goog"
}
----

* **Response Payload (201 Created):**
+
[source,json]
----
{
  "originalUrl": "https://www.google.com",
  "shortUrl": "http://localhost:8080/custom-goog",
  "shortCode": "custom-goog",
  "createdAt": "2026-07-15T15:49:25Z",
  "custom": true
}
----

=== 2. Retrieve Analytics
* **URL:** `/api/v1/analytics/{shortCode}`
* **Method:** `GET`
* **Response Payload (200 OK):**
+
[source,json]
----
{
  "shortCode": "custom-goog",
  "originalUrl": "https://www.google.com",
  "totalClicks": 1,
  "recentClicks": [
    {
      "ipAddress": "127.0.0.1",
      "userAgent": "Mozilla/5.0 ...",
      "referer": "Direct",
      "clickedAt": "2026-07-15T15:50:02Z"
    }
  ]
}
----

== How To Run the Application

=== Prerequisites
* Java 17 JDK
* Docker Desktop (for running PostgreSQL)

=== Step 1: Start PostgreSQL Database
[source,bash]
----
docker run --name local-postgres \
  -e POSTGRES_DB=url_shortener \
  -e POSTGRES_USER=postgres \
  -e POSTGRES_PASSWORD=postgres \
  -p 5432:5432 \
  -d postgres:15-alpine
----

=== Step 2: Compile and Build
[source,bash]
----
./mvnw clean compile
----

=== Step 3: Run the Service
[source,bash]
----
./mvnw spring-boot:run
----
The system will automatically run Flyway migrations and host the frontend dashboard on http://localhost:8080/[http://localhost:8080/].

=== Step 4: Run Tests
[source,bash]
----
./mvnw test
----
This triggers the comprehensive suite of Base62 unit tests and MockMvc HTTP integration tests.
